# Authentication Architecture & Security Specifications

This document outlines the authentication mechanisms supported by Incident Commander, detailing how they are implemented and the security considerations that ensure they remain robust against attacks.

## Supported Authentication Modes

Incident Commander supports multiple authentication modes to accommodate different deployment environments and security requirements. The active mode is determined by the `AuthMode` configuration (e.g., `basic`, `kratos`, `clerk`).

### 1. Basic Authentication (`htpasswd`)
*   **Description:** A straightforward authentication method using a traditional `.htpasswd` file.
*   **Implementation:** Powered by the `tg123/go-htpasswd` library.
*   **Security Rationale:**
    *   **Delegated Hashing:** Relies on the proven `.htpasswd` format, which natively supports strong, salted hashing algorithms such as bcrypt or APR1-MD5.
    *   **No Database Storage:** Passwords are never stored in the application's primary database, separating credentials from application data.

### 2. Ory Kratos
*   **Description:** Integration with Ory Kratos, an open-source, API-first identity and user management system.
*   **Implementation:** The application interacts with Kratos via its REST APIs to validate sessions and manage identities.
*   **Security Rationale:**
    *   **Offloaded Responsibility:** Identity management, password hashing, and login flows (including multi-factor authentication, account recovery, etc.) are entirely offloaded to a purpose-built, hardened identity provider.
    *   **Stateless Session Validation:** Uses Kratos session cookies or bearer tokens verified against the Kratos backend, ensuring the application itself does not handle raw credentials.

### 3. Clerk
*   **Description:** Integration with Clerk, a modern identity and user management platform.
*   **Implementation:** Validates Clerk-issued JWT session tokens (`__session` cookie or Bearer token) using Clerk's JSON Web Key Set (JWKS).
*   **Security Rationale:**
    *   **Cryptographic Verification:** Session tokens are cryptographically verified using public keys retrieved from Clerk's JWKS endpoint.
    *   **Zero-Trust Credentials:** Like Kratos, Clerk handles all sensitive credential material. The application only receives verifiable assertions of identity.

## Access Tokens (API Keys)

*   **Description:** Long-lived access tokens used primarily for automated agents, CLI access, or API integrations.
*   **Implementation:** Custom tokens generated securely and stored using Argon2id.
*   **Security Rationale:**
    *   **Argon2id Hashing:** When an access token is created, the raw token password is never stored in plaintext. Instead, it is hashed using **Argon2id**, a memory-hard and compute-hard key derivation function.
    *   **Offline Attack Resistance:** By tuning Argon2id parameters (time cost, memory cost, parallelism), the system is highly resistant to offline brute-force and dictionary attacks using specialized hardware (GPUs/ASICs).
    *   **Stateless Token Format:** The raw token provided to the user encapsulates the parameters needed to verify it (`password.salt.timeCost.memoryCost.parallelism`), while only the `encodedHash` resides in the database.

## Embedded OIDC Provider

*   **Description:** An embedded OpenID Connect (OIDC) provider built on `zitadel/oidc/v3`, allowing the application to act as an Identity Provider for CLI tools and other clients.
*   **Implementation:** Mounts standard OIDC endpoints (e.g., `/.well-known/openid-configuration`, `/oidc/login`, `/oidc/token`).
*   **Security Rationale:**
    *   **Secure Internal Encryption:** Internal OIDC state (such as cookies) is encrypted using a cryptographically secure 32-byte AES key generated via `crypto/rand`. This key is stored securely alongside the RSA signing keys, preventing attackers from forging or decrypting authorization states.
    *   **Token Hashing:** Highly sensitive, long-lived tokens such as **Refresh Tokens** and short-lived **Authorization Codes** are generated as high-entropy UUIDs. Before being persisted to the database, they are hashed using **SHA-256**.
        *   *Why SHA-256 instead of Argon2id here?* Because these tokens are generated by the server with extreme entropy (UUIDv4), they are immune to guessing or dictionary attacks. SHA-256 provides fast, deterministic hashing to prevent token leakage in the event of a database compromise, without incurring the heavy CPU/Memory penalty of Argon2id on every API request.
    *   **RSA Signing Keys:** ID Tokens are signed using RSA keys. If a signing key is not provided, the system securely generates a 2048-bit RSA key pair at startup and persists it.

## Defense in Depth Summary

The authentication architecture employs a defense-in-depth strategy:
1.  **Never Store Plaintext Secrets:** Whether it's passwords (htpasswd/Kratos), API Keys (Argon2id), or OIDC Tokens (SHA-256), the database only holds cryptographic hashes.
2.  **Right Tool for the Job:** Uses Argon2id for low-entropy or human-handled secrets (API keys) to resist brute forcing, and uses fast SHA-256 hashing for high-entropy, machine-generated tokens (OIDC).
3.  **Delegated Identity:** Wherever possible, complex identity flows are delegated to specialized systems (Kratos, Clerk).
